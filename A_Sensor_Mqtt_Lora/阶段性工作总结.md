# DHTC12 温湿度读取阶段性工作总结

## 1. 当前目标

本阶段目标是先解决 STM32F103C8T6 通过串口稳定输出 DHTC12 温湿度数据的问题，暂不接入云平台。

## 2. 硬件与协议确认

根据 DHTC12 出厂技术文件确认：

- DHTC12 使用标准 I2C 协议。
- 7 位设备地址为 `0x44`。
- 同时测量温湿度命令为 `0x2C10`。
- 软复位命令为 `0x30A2`。
- 每次读取 6 字节数据：温度高字节、温度低字节、温度 CRC、湿度高字节、湿度低字节、湿度 CRC。
- CRC 使用 CRC-8，初始值 `0xFF`，多项式 `0x31`。

当前接线约定：

| DHTC12 | STM32F103 |
| --- | --- |
| VDD | 3.3V |
| GND | GND |
| SCK / SCL | PB6 |
| SDA | PB7 |

串口调试：

| 功能 | 引脚 / 参数 |
| --- | --- |
| USART1_TX | PA9 |
| USART1_RX | PA10 |
| 波特率 | 115200 |
| 数据格式 | 8N1 |

## 3. 已定位的问题

早期现象是串口不能稳定输出温湿度，且错误信息不够明确。原驱动只能粗略输出 `I2C err` 或 `CRC err`，无法判断是地址无响应、发送命令失败、接收失败、CRC 错误，还是湿度校准参数读取失败。

此外，DHTC12 说明书中推荐上电后进行软复位，并可读取 HumA/HumB 湿度校准参数。原代码未完整体现这些初始化流程。

## 4. 本阶段修改内容

### 4.1 `Core/Inc/dht12.h`

新增 DHTC12 状态码：

- `DHTC12_OK`
- `DHTC12_ERR_NO_ACK`
- `DHTC12_ERR_TX`
- `DHTC12_ERR_RX`
- `DHTC12_ERR_CRC`
- `DHTC12_ERR_CALIB`

同时在数据结构中增加：

- `hal_error`：记录 HAL I2C 错误码。
- `used_calibration`：记录本次湿度计算是否使用校准参数。

### 4.2 `Core/Src/dht12.c`

驱动改为更完整的 DHTC12 HAL I2C 实现：

- 上电后探测 `0x44` 地址是否有 ACK。
- 执行软复位命令 `0x30A2`。
- 尝试读取 HumA/HumB 校准参数。
- 发送温湿度测量命令 `0x2C10`。
- 延时等待测量完成后读取 6 字节数据。
- 分别校验温度和湿度 CRC。
- 优先使用 HumA/HumB 计算湿度；校准参数不可用时回退到线性换算。

### 4.3 `Core/Src/main.c`

串口输出改为更明确：

启动时打印 DHTC12 初始化结果：

```text
DHTC12 Init OK, cal=YES
```

或：

```text
DHTC12 Init OK, cal=NO, using linear humidity fallback
```

循环读取成功时打印：

```text
Temp: 23.9C, Hum: 35.3%, cal=NO
```

读取失败时打印具体错误：

```text
DHTC12 NO_ACK, hal_error=0x00000000
DHTC12 TX_ERR, hal_error=0x...
DHTC12 RX_ERR, hal_error=0x...
DHTC12 CRC_ERR, hal_error=0x..., raw=...
```

说明 STM32 已能通过 I2C 读取 DHTC12，并通过 USART1 稳定输出温湿度数据。

## 6. 当前状态说明

`cal=NO` 表示 HumA/HumB 校准参数读取未成功，当前使用线性湿度换算作为 fallback。温湿度读取主链路已经跑通，后续可以继续研究 HumA/HumB 寄存器读取时序或直接保留当前方案用于实训展示。

## 7. 下一阶段建议

1. 整理接线图，固定 DHTC12 与 STM32 的连接方式。
2. 继续观察串口温湿度数据是否随环境变化而变化。
3. 若需要更高湿度精度，再单独调试 HumA/HumB 校准参数读取。
4. 在串口读取稳定后，再把该驱动接回 ESP8266 和 OneNet 云平台上报流程。
